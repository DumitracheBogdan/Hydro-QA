# Parity Contract — Per-Datum Dual-UI Evidence

**Date:** 2026-05-28 · **Drives:** `scripts/parity/webapp-shots.mjs` (to build) · **Inputs:** `docs/PARITY-FACTS.md`, `docs/API-MAP-BE.md`, `webapp-baseline/pages.json` (captured visit-details DOM).

This is the machine-readable contract that pins, for **each shared datum**, (1) the BE field, (2) the API connection check, (3) where/how it is **set** on the webapp, (4) where it is **displayed** on the webapp (for the verify screenshot), (5) the mobile selector + strategy, and (6) the **evidence mode** chosen under the approved HYBRID decision.

Evidence modes (HYBRID decision):

| Mode | Meaning |
|---|---|
| `real-web-set+mobile-verify` | Web field is **editable in the webapp UI**: type + Save on the real webapp, screenshot the set, then verify the value appears on mobile. (web→mobile, "real" path) |
| `api-set+web-screenshot+mobile-verify` | Web field is **read-only / display-only** for this web→mobile datum: API-PATCH the value, screenshot the webapp **displaying** it, verify it on mobile. |
| `mobile-set+web-render-verify` | mobile→web: set the value on mobile (Maestro), then screenshot the webapp **rendering** it. |
| `api-api-only` | API create→read tautology only (no cross-platform UI render). Used for **2c** per gap **F-01**. |

> **Direction is the source-of-truth axis; evidence mode is the method.** `2c` keeps `direction = web→mobile` even though its mode is `api-api-only` — the `api` token lives only in `evidenceMode`, never in `direction`.

## Notes

### FROZEN_CLOCK caveat (must un-freeze for live parity)
`scripts/lib/webapp-login.mjs` exports `FROZEN_CLOCK = 2026-04-16T09:00:00Z` and `launchAuthed()` calls `page.clock.setFixedTime(FROZEN_CLOCK)`. But `setup-data.mjs` creates the run's visit at **real `new Date()`** (today, scheduled `now + 2h`). Under the frozen clock the webapp believes "today" is **mid-April 2026**, so the freshly-created visit is ~6 weeks in the **future** and falls **outside** the default visit list / calendar windows → `webapp-shots.mjs` will not find the run's visit to screenshot (it lands on an empty/wrong day). **`webapp-shots.mjs` must run on REAL time** (do not call `setFixedTime`, or fix the clock to the visit's actual `from` date). The frozen clock is fine for static UI-baseline diffing but is wrong for live, freshly-seeded parity data. Reach the visit directly by URL (`/visits/details/{visitId}` from `parity-context.json`) to sidestep date-window filtering entirely where possible.

### Field labels are backend-config (not source)
All form `formName` / `fieldName` strings (`Visit Information`, `Risk Assessment`, `Assisting 1/2/3`, `Works being carried out`, `Accessing Area/Lone Working- Comments`, `Site Induction required & Completed`, dropdown options) are **runtime data in the `form_fields` table**, configured per job type — **not hardcoded in TypeScript** (`API-MAP-BE.md`: "There is NO hardcoded key path"). The exact strings used here are the **empirically verified** labels for jobType `658f27c1-9306-42a2-81a6-ad249d7eaef3` ("Health and Safty Risk Assessment"), taken from `PARITY-FACTS.md`. If the form config changes, re-discover via `GET /form-fields?jobTypeId=658f27c1...` and update both the flows and `buildExpected()` together (they consume shared constants in `setup-data.mjs` so they cannot drift).

### 2d / 2g editability flip-condition
`UpdateVisitDto` / `UpdateInspectionDto` accept `waterSystemDescription` / `itemDetail` (PATCH → 200), so the webapp **edit** form *may* expose them as editable. Nothing in the docs or the static baseline DOM settles this. Until confirmed against the live dev webapp (login → open the run's visit/inspection edit form → check for an editable input), 2d/2g ship as `api-set+web-screenshot+mobile-verify`. **If a future live pass finds the inputs editable, flip 2d/2g `webSetSelector` to the real input + Save and `evidenceMode` to `real-web-set+mobile-verify`.** This is the single unresolved item; it does not block writing the contract, but the `api-set` mode requires the named web display widget (the expandable "Visit Details" card / Inspections tab) to actually render the value for the screenshot to be meaningful.
